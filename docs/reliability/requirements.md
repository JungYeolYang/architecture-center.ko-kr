# <a name="developing-requirements-for-resilient-azure-applications"></a>Azure 복원 력 있는 응용 프로그램에 대 한 요구 사항 개발

구축 *복원 력* (오류 로부터 복구) 및 *가용성* (상당한 가동 중지 없이 정상 상태에서 실행)를 앱에 요구 사항 수집으로 시작 합니다. 예를 들어 가동 중지 시간이 얼마나 적합? 잠재적인 가동 중지 시간에 비즈니스가 비용은 얼마 인가요? 고객의 가용성 요구 사항은 무엇입니까? 응용 프로그램을 항상 사용할 수 있도록에 얼마나 투자 수행? 비용과 위험 란?

## <a name="identify-distinct-workloads"></a>고유한 워크 로드 식별

일반적으로 여러 응용 프로그램 클라우드 솔루션에 이루어진 *워크 로드*합니다. 워크 로드에는 고유한 기능 또는 비즈니스 논리와 데이터 저장소 요구 사항 측면에서 다른 태스크와에서 논리적으로 분리 된 작업입니다. 예를 들어 전자 상거래 앱에는 다음 워크 로드는 있을 수 있습니다.

- 제품 카탈로그를 찾아보고 검색합니다.
- 주문을 작성하고 추적합니다.
- 추천 제품을 살펴봅니다.

각 워크 로드에는 가용성, 확장성, 데이터 일관성 및 재해 복구에 대 한 다른 요구 사항이 있습니다. 각 워크 로드에 대 한 위험 및 비용을 분산 하 여 비즈니스 결정을 확인 합니다.

또한 서비스 수준 목표 워크 로드를 분해 합니다. 서비스 구성 된 경우 중요 하 고 덜 중요 워크 로드를 다르게 관리 하도록 하 고 서비스 기능 및 가용성 요구 사항을 충족 하는 데 필요한 인스턴스 수를 지정 합니다.

## <a name="plan-for-usage-patterns"></a>사용 패턴에 대 한 계획

위험 및 중요 하지 않은 기간 동안 요구 사항의 차이점을 식별 합니다. 시스템이 반드시 작동해야 하는 특정 중요 시간대가 있습니까? 예를 들어 세금 신고 응용 프로그램을 신고 마감 하는 동안 실패 수 없습니다 하 고 비디오 스트리밍 서비스는 라이브 이벤트 동안 지연 해서는 안 됩니다. 이러한 상황에서 하는 비용과 위험 합니다.

- 가동 시간을 보장 하 고 중요 한 기간에 서비스 수준 계약 (Sla)을 충족 하려면 중복성을 계획 여러 지역에 걸쳐 하나가 실패할 경우 많은 비용이 소요 하는 경우에 합니다.
- 반대로, 중요 하지 않은 기간 동안 비용을 최소화 하기 위해 단일 지역에서 응용 프로그램을 실행 합니다.
- 경우에 따라 사용량 기반 요금 청구는 최신 서버 리스 기술을 사용 하 여 추가 비용을 줄일 수 있습니다.

## <a name="establish-availability-and-recovery-metrics"></a>가용성 및 복구 메트릭을 설정 합니다.

요구 사항 프로세스의 일부로 메트릭의 두 집합에 대 한 기준 수치를 만듭니다. 첫 번째 집합을 사용 하 여 클라우드 서비스에 중복성을 추가 하는 위치와 고객에 게 제공 하는 Sla를 확인할 수 있습니다. 두 번째 재해 복구를 계획할 수 있습니다 설정 합니다.

### <a name="availability-metrics"></a>가용성 메트릭

이러한 측정값을 사용 하 여 중복 계획 하 고 고객 Sla 확인.

- **Mttr (평균)를 복구 하는 평균 시간은** 는 평균 시간 오류가 발생 한 후 구성 요소를 복원 하는 데 걸리는 합니다.
- **(Mtbf) 간 평균 시간** 는 방법이 긴 구성 요소를 예상할 수 중단 간에 지속 되도록 합니다.

### <a name="recovery-metrics"></a>복구 메트릭

위험 평가 수행 하 여 이러한 값을 파생 하 고 비용 및 가동 중지 시간과 데이터 손실의 위험을 이해 해야 합니다. 이러한 시스템의 비기능적 요구 사항이 되며 비즈니스 요구 사항에 따라 결정 해야 합니다.

- **복구 시간 목표 (RTO)** 사고 발생 후 응용 프로그램을 사용할 수 없는 최대 허용 가능한 시간입니다.
- **복구 지점 목표 (RPO)** 재해 발생 시 허용 되는 데이터 손실의 최대 지속 시간입니다.

경우 MTTR 값 *모든* 시스템 RTO를 초과 하는 항상 사용 가능한 설치의 핵심 구성 요소, 시스템 오류는 허용 되지 않는 업무 중단이 발생할 수 있습니다. 즉, 정의 된 RTO 내에서 시스템을 복원할 수 없습니다.

## <a name="determine-workload-availability-targets"></a>워크 로드 가용성 목표를 결정 합니다.

아키텍처는 비즈니스 요구 사항을 충족 하는지 확인할 수 있도록 솔루션에서 각 워크 로드에 대 한 고유의 목표 Sla를 정의 합니다.

### <a name="consider-cost-and-complexity"></a>비용 및 복잡성을 고려

다른 모든 항목을 더 높은 가용성 것이 좋습니다. 그러나 비용 및 복잡성이 증가 자세한 99.999 9에 따른 합니다. 가동 시간 99.99%의 월간 총 가동 중지 시간을 약 5 분으로 변환 됩니다. 복잡성과 99.999 도달 하는 비용을 들일 가치가 있습니까? 대답은 비즈니스 요구 사항에 따라 다릅니다.

다음은 SLA를 정의할 때 고려해야 할 다른 사항입니다.

- 4 개의 9 (99.99%)를 위해 오류 로부터 복구 하려면 수동 개입에 사용할 수 없게 합니다. 애플리케이션이 자체적으로 진단하고 자체적으로 복구해야 합니다.
- 99.99%를 초과 SLA를 충족할 만큼 신속 하 게 가동 중단을 감지 하 어렵습니다.
- SLA가 측정되는 시간을 생각해 보세요. 시간이 짧을수록 허용 오차도 작습니다. 시간별 또는 일별 작동 시간 측면에서 SLA를 정의 하려면 적합 하지 않습니다.
- MTBF 및 MTTR 측정값을 고려해야 합니다. 높을수록 SLA, 덜 자주 서비스가 중단 될 수 있습니다 하 고 서비스를 더 빠르게 복구 해야 합니다.
- 응용 프로그램 각 부분의 가용성 목표에 대 한 고객의 계약 가져오기 및 문서화 합니다. 이 고, 그렇지 디자인 고객의 기대를 충족 하지 않을 수 있습니다.

### <a name="identify-dependencies"></a>종속성 확인

내부 및 외부 종속성을 식별 하려면 종속성 매핑을 연습을 수행 합니다. 예제에서는 보안 또는 Active Directory 또는 결제 공급자와 같은 타사 서비스와 같은 id와 관련 된 종속성을 포함 또는 전자 메일 메시징 서비스입니다.

오류의 단일 지점 또는 병목 현상이 발생할 수 있는 외부 종속성에 특히 주의 합니다. 작업 필요 99.99% 가동 시간 99.9 %SLA 사용 하 여 서비스에 따라 달라 집니다 하지만 해당 서비스 시스템을 단일 실패 지점이 일 수 없습니다. 한 가지 해결책은 대체 경로 서비스가 실패 하는 경우입니다. 또는 해당 서비스의 오류를 복구 하려면 다른 조치를 수행 합니다.

## <a name="understand-service-level-agreements"></a>서비스 수준 계약을 이해

Azure에서의 [서비스 수준 계약](https://azure.microsoft.com/en-us/support/legal/sla/) 작동 시간 및 연결에 대 한 Microsoft의 약속을 설명 합니다. 특정 서비스에 대 한 SLA는 99.9% 인 경우에 99.9%의 시간 동안 사용 가능 하도록 서비스를 하시면 됩니다. 다른 서비스에 다른 Sla.

Azure SLA에 또한 서비스 크레딧의 구체적인 정의와 함께 SLA가 충족 되지 않으면 가져오기에 대 한 프로 비전 *가용성* 각 서비스에 대 한 합니다. SLA의 이러한 측면은 적용 정책의 역할을 합니다.

### <a name="composite-slas"></a>복합 SLA

*복합 Sla* 서로 다른 수준의 가용성을 사용 하 여 각 응용 프로그램을 지 원하는 여러 서비스가 관련 되어 있습니다. Azure SQL Database에 쓰는 App Service 웹 앱을 예로 들어 보겠습니다. 이 문서가 작성된 시점에 이러한 Azure 서비스의 SLA는 다음과 같습니다.

- App Service 웹 앱 = 99.95%
- SQL Database = 99.99%

이 애플리케이션에 기대하는 최대 가동 중지 시간이 얼마입니까? 서비스 중 하나가 중단되면 전체 애플리케이션이 중단됩니다. 각 서비스 실패의 확률은 독립적 이므로이 응용 프로그램에 대 한 복합 SLA는 99.95% × 99.99% = 99.94%입니다. 여러 서비스를 사용 하는 응용 프로그램은 잠재적 오류 지점이 더 때문에 놀라운 아닌 개별 Sla 보다 작습니다.

독립적인 대체 경로 만들어서 복합 SLA를 높일 수 있습니다. 예를 들어 SQL Database를 사용할 수 없는 트랜잭션을 나중에 처리 되도록 큐에 배치 합니다.

![복합 SLA](_images/composite-sla.png)

이 디자인에서는 애플리케이션이 데이터베이스에 연결할 수 없는 경우에도 계속 사용할 수 있습니다. 그러나 데이터베이스와 큐가 동시에 중단되면 응용 프로그램도 중단됩니다. 예상 되는 동시에 오류에 대 한 시간 비율은 0.0001 × 0.001 이므로이 복합 경로의 복합 SLA는:

- 데이터베이스 *또는* 큐 = 1.0 − (0.0001 × 0.001) = 99.99999%

총 복합 SLA는 다음과 같습니다.

- 웹 앱 *하 고* (데이터베이스 *하거나* 큐) = 99.95% × 99.99999% = \~99.95%

도이 방식의 장단점이 있습니다. 응용 프로그램 논리는 복잡 하 고 큐에 대해 지불 하는 데이터 일관성 문제를 고려해 야 합니다.

### <a name="slas-for-multiregion-deployments"></a>다중 지역 배포에 대 한 Sla

다중 지역 배포의 복합 SLA는 다음과 같이 계산 됩니다.

- *N* 한 지역에서 배포 응용 프로그램에 대 한 복합 SLA는 합니다.

- 두 지역에 대 한 결합 된 SLA = ((1 − 0.9995) 1 − \^ 2) = 99.999975%
- 4 개 지역에 대 한 결합 된 SLA = ((1 − 0.9995) 1 − \^ 4) 99.999999% =
