## <a name="how-to-fix-the-problem"></a><span data-ttu-id="8af3d-124">문제를 해결하는 방법</span><span class="sxs-lookup"><span data-stu-id="8af3d-124">How to fix the problem</span></span>

<span data-ttu-id="8af3d-125">동기 I/O 작업을 비동기 작업으로 바꿉니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-125">Replace synchronous I/O operations with asynchronous operations.</span></span> <span data-ttu-id="8af3d-126">그러면 현재 스레드가 해제되어 차단되지 않고 의미 있는 작업을 계속 수행할 수 있어 계산 리소스 사용률이 개선됩니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-126">This frees the current thread to continue performing meaningful work rather than blocking, and helps improve the utilization of compute resources.</span></span> <span data-ttu-id="8af3d-127">I/O를 비동기로 수행하는 방법은 클라이언트 응용 프로그램에서의 예기치 않은 요청 급증을 처리할 때 특히 효율적입니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-127">Performing I/O asynchronously is particularly efficient for handling an unexpected surge in requests from client applications.</span></span> 

<span data-ttu-id="8af3d-128">많은 라이브러리는 메서드의 동기 버전과 비동기 버전을 모두 제공합니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-128">Many libraries provide both synchronous and asynchronous versions of methods.</span></span> <span data-ttu-id="8af3d-129">가능하면 언제나 비동기 버전을 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-129">Whenever possible, use the asynchronous versions.</span></span> <span data-ttu-id="8af3d-130">다음은 Azure Blob Storage에 파일을 업로드하는 앞의 예의 비동기 버전입니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-130">Here is the asynchronous version of the previous example that uploads a file to Azure blob storage.</span></span>

<span data-ttu-id="8af3d-136">작업의 비동기 버전을 제공하지 않는 버전의 경우 선택한 동기 방법의 주위에 비동기 래퍼를 만들 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-136">For libraries that don't provide asynchronous versions of operations, it may be possible to create asynchronous wrappers around selected synchronous methods.</span></span> <span data-ttu-id="8af3d-137">이 접근법은 주의해서 따라야 합니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-137">Follow this approach with caution.</span></span> <span data-ttu-id="8af3d-138">이 접근법은 비동기 래퍼를 호출하는 스레드 응답성을 개선할 수 있지만 실제로는 더 많은 리소스를 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-138">While it may improve responsiveness on the thread that invokes the asynchronous wrapper, it actually consumes more resources.</span></span> <span data-ttu-id="8af3d-139">추가 스레드가 만들어질 수 있으며 이 스레드가 수행한 작업의 동기화와 연결된 오버헤드가 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-139">An extra thread may be created, and there is overhead associated with synchronizing the work done by this thread.</span></span> <span data-ttu-id="8af3d-140">다음 블로그 게시물에서 몇 가지 절충안을 설명합니다. ## <a name="considerations"></a><span data-ttu-id="8af3d-143">고려 사항</span><span class="sxs-lookup"><span data-stu-id="8af3d-143">Considerations</span></span>

- <span data-ttu-id="8af3d-144">매우 수명이 짧아서 경합을 야기할 가능성이 낮은 I/O 작업은 비동기 작업으로서 더 효율적일 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-144">I/O operations that are expected to be very short lived and are unlikely to cause contention might be more performant as synchronous operations.</span></span> <span data-ttu-id="8af3d-145">SSD 드라이브에 있는 작은 파일을 읽는 경우가 그러한 예일 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-145">An example might be reading small files on an SSD drive.</span></span> <span data-ttu-id="8af3d-146">작업이 완료될 때 작업을 다른 스레드에 디스패치하고 해당 스레드와 동기화하는 오버헤드가 비동기 I/O의 이점보다 클 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-146">The overhead of dispatching a task to another thread, and synchronizing with that thread when the task completes, might outweigh the benefits of asynchronous I/O.</span></span> <span data-ttu-id="8af3d-147">그러나 이러한 경우는 비교적 드물며 대부분의 I/O 작업은 비동기로 수행해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-147">However, these cases are relatively rare, and most I/O operations should be done asynchronously.</span></span>

- <span data-ttu-id="8af3d-148">I/O 성능이 개선되면 시스템의 다른 부분에 병목이 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-148">Improving I/O performance may cause other parts of the system to become bottlenecks.</span></span> <span data-ttu-id="8af3d-149">예를 들어 스레드를 차단 해제하면 동시 공유 리소스 요청이 더 많아지고 그에 따라 리소스 부족 또는 제한이 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-149">For example, unblocking threads might result in a higher volume of concurrent requests to shared resources, leading in turn to resource starvation or throttling.</span></span> <span data-ttu-id="8af3d-150">이러한 상황이 문제가 된다면 경합을 줄이기 위해 웹 서버 수를 규모 확장하거나 데이터 저장소를 분할해야 할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="8af3d-150">If that becomes a problem, you might need to scale out the number of web servers or partition data stores to reduce contention.</span></span>
