# <a name="extraneous-fetching-antipattern"></a><span data-ttu-id="a0ec0-103">불필요한 가져오기 안티패턴</span><span class="sxs-lookup"><span data-stu-id="a0ec0-103">Extraneous Fetching antipattern</span></span>

## <a name="problem-description"></a><span data-ttu-id="a0ec0-105">문제 설명</span><span class="sxs-lookup"><span data-stu-id="a0ec0-105">Problem description</span></span>

<span data-ttu-id="a0ec0-106">이 안티패턴은 애플리케이션이 필요*할 수 있는* 모든 데이터를 검색하여 I/O 요청의 최소화를 시도하는 경우 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-106">This antipattern can occur if the application tries to minimize I/O requests by retrieving all of the data that it *might* need.</span></span> <span data-ttu-id="a0ec0-107">이는 흔히 [번잡한 I/O][chatty-io] 안티패턴을 과도하게 보정한 결과입니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-107">This is often a result of overcompensating for the [Chatty I/O][chatty-io] antipattern.</span></span> <span data-ttu-id="a0ec0-108">예를 들어 응용 프로그램이 데이터베이스의 모든 제품에 대한 세부정보를 가져올 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-108">For example, an application might fetch the details for every product in a database.</span></span> <span data-ttu-id="a0ec0-109">그러나 사용자는 해당 세부정보의 일부만 필요하며(일부는 고객에게 유용하지 않을 수 있음) 아마도 *모든* 제품을 동시에 볼 필요가 없을 것입니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-109">But the user may need just a subset of the details (some may not be relevant to customers), and probably doesn't need to see *all* of the products at once.</span></span> <span data-ttu-id="a0ec0-110">사용자가 전체 카탈로그를 탐색한다 하더라도 결과를 한 번에 &mdash; 20개씩 보여 주는 페이지로 나누는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-110">Even if the user is browsing the entire catalog, it would make sense to paginate the results &mdash; showing 20 at a time, for example.</span></span>

<span data-ttu-id="a0ec0-111">이 문제의 또 다른 출처는 좋지 않은 프로그래밍 또는 설계 관행을 따르는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-111">Another source of this problem is following poor programming or design practices.</span></span> <span data-ttu-id="a0ec0-112">예를 들어 다음 코드는 Entity Framework를 사용하여 모든 제품에 전체 세부정보를 가져옵니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-112">For example, the following code uses Entity Framework to fetch the complete details for every product.</span></span> <span data-ttu-id="a0ec0-113">그런 다음 결과를 필터링하여 필드의 일부만 반환하고 나머지는 버립니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-113">Then it filters the results to return only a subset of the fields, discarding the rest.</span></span> <span data-ttu-id="a0ec0-114">전체 샘플은 [여기][sample-app]에서 찾을 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-114">You can find the complete sample [here][sample-app].</span></span>

```csharp
public async Task<IHttpActionResult> GetAllFieldsAsync()
{
    using (var context = new AdventureWorksContext())
    {
        // Execute the query. This happens at the database.
        var products = await context.Products.ToListAsync();

        // Project fields from the query results. This happens in application memory.
        var result = products.Select(p => new ProductInfo { Id = p.ProductId, Name = p.Name });
        return Ok(result);
    }
}
```

<span data-ttu-id="a0ec0-115">다음 예제에서 응용 프로그램은 데이터베이스 대신 수행할 수 있는 집계를 수행합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-115">In the next example, the application retrieves data to perform an aggregation that could be done by the database instead.</span></span> <span data-ttu-id="a0ec0-116">이 응용 프로그램은 판매한 모든 주문에 대한 모든 기록을 가져온 다음 해당 레코드에 대한 합계를 계산하여 총 매출을 계산합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-116">The application calculates total sales by getting every record for all orders sold, and then computing the sum over those records.</span></span> <span data-ttu-id="a0ec0-117">전체 샘플은 [여기][sample-app]에서 찾을 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-117">You can find the complete sample [here][sample-app].</span></span>

```csharp
public async Task<IHttpActionResult> AggregateOnClientAsync()
{
    using (var context = new AdventureWorksContext())
    {
        // Fetch all order totals from the database.
        var orderAmounts = await context.SalesOrderHeaders.Select(soh => soh.TotalDue).ToListAsync();

        // Sum the order totals in memory.
        var total = orderAmounts.Sum();
        return Ok(total);
    }
}
```

<span data-ttu-id="a0ec0-118">다음 예제에서는 Entity Framework가 LINQ to Entities를 사용하는 방법 때문에 발생하는 미묘한 문제를 보여 줍니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-118">The next example shows a subtle problem caused by the way Entity Framework uses LINQ to Entities.</span></span> 

```csharp
var query = from p in context.Products.AsEnumerable()
            where p.SellStartDate < DateTime.Now.AddDays(-7) // AddDays cannot be mapped by LINQ to Entities
            select ...;

List<Product> products = query.ToList();
```

<span data-ttu-id="a0ec0-119">응용 프로그램은 `SellStartDate`주 이상 오래된 제품을 찾으려고 합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-119">The application is trying to find products with a `SellStartDate` more than a week old.</span></span> <span data-ttu-id="a0ec0-120">대부분의 경우 LINQ to Entities는 `where` 절을 데이터베이스에서 실행되는 SQL 문으로 변환합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-120">In most cases, LINQ to Entities would translate a `where` clause to a SQL statement that is executed by the database.</span></span> <span data-ttu-id="a0ec0-121">그러나 이 경우 LINQ to Entities는 `AddDays` 메서드를 SQL에 매핑할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-121">In this case, however, LINQ to Entities cannot map the `AddDays` method to SQL.</span></span> <span data-ttu-id="a0ec0-122">그 대신에 `Product` 테이블의 모든 행이 반환되며 그 결과가 메모리에서 필터링됩니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-122">Instead, every row from the `Product` table is returned, and the results are filtered in memory.</span></span> 

<span data-ttu-id="a0ec0-123">`AsEnumerable` 호출이 바로 문제가 있다는 힌트입니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-123">The call to `AsEnumerable` is a hint that there is a problem.</span></span> <span data-ttu-id="a0ec0-124">이 메서드는 결과를 `IEnumerable` 인터페이스로 변환합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-124">This method converts the results to an `IEnumerable` interface.</span></span> <span data-ttu-id="a0ec0-125">`IEnumerable`은 필터링을 지원하지만 이 필터링은 데이터베이스가 아닌 *클라이언트* 쪽에서 수행됩니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-125">Although `IEnumerable` supports filtering, the filtering is done on the *client* side, not the database.</span></span> <span data-ttu-id="a0ec0-126">기본적으로 LINQ to Entities는 `IQueryable`을 사용하여 필터링 책임을 데이터 원본에 넘깁니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-126">By default, LINQ to Entities uses `IQueryable`, which passes the responsibility for filtering to the data source.</span></span> 

## <a name="how-to-fix-the-problem"></a><span data-ttu-id="a0ec0-127">문제를 해결하는 방법</span><span class="sxs-lookup"><span data-stu-id="a0ec0-127">How to fix the problem</span></span>

<span data-ttu-id="a0ec0-128">금세 오래되거나 삭제될 수 있는 대량의 데이터를 가져오지 않고 수행 중인 작업에 필요한 데이터만 가져옵니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-128">Avoid fetching large volumes of data that may quickly become outdated or might be discarded, and only fetch the data needed for the operation being performed.</span></span> 

<span data-ttu-id="a0ec0-129">테이블의 모든 열을 가져와서 필터링하는 대신에 데이터베이스에서 필요한 열을 선택합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-129">Instead of getting every column from a table and then filtering them, select the columns that you need from the database.</span></span>

```csharp
public async Task<IHttpActionResult> GetRequiredFieldsAsync()
{
    using (var context = new AdventureWorksContext())
    {
        // Project fields as part of the query itself
        var result = await context.Products
            .Select(p => new ProductInfo {Id = p.ProductId, Name = p.Name})
            .ToListAsync();
        return Ok(result);
    }
}
```

<span data-ttu-id="a0ec0-130">마찬가지로 응용 프로그램 메모리가 아닌 데이터베이스에서 집계를 수행합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-130">Similarly, perform aggregation in the database and not in application memory.</span></span>

```csharp
public async Task<IHttpActionResult> AggregateOnDatabaseAsync()
{
    using (var context = new AdventureWorksContext())
    {
        // Sum the order totals as part of the database query.
        var total = await context.SalesOrderHeaders.SumAsync(soh => soh.TotalDue);
        return Ok(total);
    }
}
```

<span data-ttu-id="a0ec0-131">Entity Framework를 사용하는 경우 LINQ 쿼리가 `IEnumerable`이 아닌 `IQueryable` 인터페이스를 사용하여 확인되도록 합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-131">When using Entity Framework, ensure that LINQ queries are resolved using the `IQueryable`interface and not `IEnumerable`.</span></span> <span data-ttu-id="a0ec0-132">데이터 원본에 매핑할 수 있는 함수만 사용하여 쿼리를 조정해야 할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-132">You may need to adjust the query to use only functions that can be mapped to the data source.</span></span> <span data-ttu-id="a0ec0-133">앞의 예제를 리팩터링하여 쿼리에서 `AddDays` 메서드를 제거하면 데이터베이스에서 필터링을 수행할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-133">The earlier example can be refactored to remove the `AddDays` method from the query, allowing filtering to be done by the database.</span></span>

```csharp
DateTime dateSince = DateTime.Now.AddDays(-7); // AddDays has been factored out. 
var query = from p in context.Products
            where p.SellStartDate < dateSince // This criterion can be passed to the database by LINQ to Entities
            select ...;

List<Product> products = query.ToList();
```

## <a name="considerations"></a><span data-ttu-id="a0ec0-134">고려 사항</span><span class="sxs-lookup"><span data-stu-id="a0ec0-134">Considerations</span></span>

- <span data-ttu-id="a0ec0-135">경우에 따라 데이터를 행 분할하여 성능을 개선할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-135">In some cases, you can improve performance by partitioning data horizontally.</span></span> <span data-ttu-id="a0ec0-136">서로 다른 작업이 데이터의 서로 다른 특성에 액세스하는 경우 행 분할로 경합을 줄일 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-136">If different operations access different attributes of the data, horizontal partitioning may reduce contention.</span></span> <span data-ttu-id="a0ec0-137">흔히 대부분의 작업은 데이터의 작은 하위 집합에 대해 실행되므로 이러한 부하 분산으로 성능이 개선될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-137">Often, most operations are run against a small subset of the data, so spreading this load may improve performance.</span></span> <span data-ttu-id="a0ec0-138">[데이터 분할][data-partitioning]을 참조하세요.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-138">See [Data partitioning][data-partitioning].</span></span>

- <span data-ttu-id="a0ec0-143">집계 같은 특정 기능을 지원하지 않는 데이터 원본을 사용하는 경우 계산된 결과를 다른 곳에 저장하고 레코드가 추가되거나 업데이트될 때 값을 업데이트할 수 있으므로 응용 프로그램이 필요할 때마다 값을 다시 계산할 필요가 없습니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-143">If you're using a data store that doesn't support a particular function, such as aggregration, you could store the calculated result elsewhere, updating the value as records are added or updated, so the application doesn't have to recalculate the value each time it's needed.</span></span>

- <span data-ttu-id="a0ec0-144">요청이 다수의 필드를 검색하는 것으로 보이면 소스 코드를 검토하여 이 필드가 모두 실제로 필요한지 여부를 결정합니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-144">If you see that requests are retrieving a large number of fields, examine the source code to determine whether all of these fields are actually necessary.</span></span> <span data-ttu-id="a0ec0-145">경우에 따라 이러한 요청은 잘못 설계된 `SELECT *` 쿼리의 결과입니다.</span><span class="sxs-lookup"><span data-stu-id="a0ec0-145">Sometimes these requests are the result of poorly designed `SELECT *` query.</span></span>
