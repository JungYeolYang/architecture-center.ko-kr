<span data-ttu-id="b7bce-108">더 빠른 릴리스 주기는 마이크로 서비스 아키텍처를 채택하는 가장 큰 이유 중 하나입니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-108">Faster release cycles are one of the biggest reasons to adopt a microservices architecture.</span></span>

<span data-ttu-id="b7bce-114">마이크로 서비스 기본 원칙을 따르면 모든 팀이 줄을 서야 하는 긴 릴리스 기차가 없어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-114">Following the microservices philosophy, there should never be a long release train where every team has to get in line.</span></span> <span data-ttu-id="b7bce-115">서비스 "A"를 빌드하는 팀은 병합, 테스트 및 배포될 서비스 "B"의 변경 내용을 기다릴 필요 없이 언제든지 업데이트를 릴리스할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-115">The team that builds service "A" can release an update at any time, without waiting for changes in service "B" to be merged, tested, and deployed.</span></span> <span data-ttu-id="b7bce-116">CI/CD 프로세스는 이를 가능하게 만드는 데 중요합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-116">The CI/CD process is critical to making this possible.</span></span> <span data-ttu-id="b7bce-117">릴리스 파이프라인은 업데이트 배포의 위험이 최소화되도록 자동화되고 매우 안정적이어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-117">Your release pipeline must be automated and highly reliable, so that the risks of deploying updates are minimized.</span></span> <span data-ttu-id="b7bce-118">매일 또는 하루에 여러 번 프로덕션에 대해 릴리스하는 경우 재발 또는 서비스 중단은 매우 드물어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-118">If you are releasing to production daily or multiple times a day, regressions or service disruptions must be very rare.</span></span> <span data-ttu-id="b7bce-119">동시에 잘못된 업데이트가 배포되는 경우 이전 버전의 서비스로 롤백하거나 롤포워드하는 신뢰할 수 있는 방법이 있어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-119">At the same time, if a bad update does get deployed, you must have a reliable way to quickly roll back or roll forward to a previous version of a service.</span></span>

<span data-ttu-id="b7bce-120">CI/CD에 대해 논의할 때 실제로 여러 가지 관련된 프로세스: 지속적인 통합, 지속적인 업데이트 및 지속적인 배포에 대해 논의합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-120">When we talk about CI/CD, we are really talking about several related processes: Continuous integration, continuous delivery, and continuous deployment.</span></span>

- <span data-ttu-id="b7bce-122">지속적인 업데이트는 CI 프로세스를 전달하는 코드 변경 내용이 프로덕션 환경과 유사한 환경으로 자동으로 게시되는 것을 의미 합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-122">Continuous delivery means that code changes that pass the CI process are automatically published to a production-like environment.</span></span> <span data-ttu-id="b7bce-123">라이브 프로덕션 환경으로 배포에는 수동 승인이 필요할 수 있지만 그렇지 않은 경우 자동화됩니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-123">Deployment into the live production environment may require manual approval, but is otherwise automated.</span></span> <span data-ttu-id="b7bce-124">목표는 코드가 항상 프로덕션 환경으로 배포될 *준비*가 되어야 하는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-124">The goal is that your code should always be *ready* to deploy into production.</span></span>

- <span data-ttu-id="b7bce-125">지속적인 배포는 CI/CD 프로세스를 전달하는 코드 변경 내용이 프로덕션 환경으로 자동으로 배포되는 것을 의미합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-125">Continuous deployment means that code changes that pass the CI/CD process are automatically deployed into production.</span></span>

<span data-ttu-id="b7bce-126">Kubernetes 및 마이크로 서비스의 컨텍스트에서 CI 단계는 컨테이너 이미지를 구축 및 테스트하고 해당 이미지를 컨테이너 레지스트리로 푸시하는 것과 관련되어 있습니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-126">In the context of Kubernetes and microservices, the CI stage is concerned with building and testing container images, and pushing those images to a container registry.</span></span> <span data-ttu-id="b7bce-127">배포 단계에서 Pod 사양은 최신 프로덕션 이미지를 선택하도록 업데이트됩니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-127">In the deployment stage, pod specs are updated to pick up the latest production image.</span></span>

## <a name="challenges"></a><span data-ttu-id="b7bce-128">과제</span><span class="sxs-lookup"><span data-stu-id="b7bce-128">Challenges</span></span>

- <span data-ttu-id="b7bce-129">**다수의 작은 독립적인 코드 베이스**</span><span class="sxs-lookup"><span data-stu-id="b7bce-129">**Many small independent code bases**.</span></span> <span data-ttu-id="b7bce-130">각 팀은 자체 빌드 파이프라인으로 자체 서비스 구축을 담당합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-130">Each team is responsible for building its own service, with its own build pipeline.</span></span> <span data-ttu-id="b7bce-131">일부 조직에서 팀은 별도 코드 리포지토리를 사용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-131">In some organizations, teams may use separate code repositories.</span></span> <span data-ttu-id="b7bce-132">이로 인해 시스템을 구축하는 방법에 대한 정보가 팀 간에 분산되고 전체 응용 프로그램을 배포하는 방법을 알고 있는 사람이 조직에서 아무도 없는 상황이 발생할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-132">This could lead to a situation where the knowledge of how to build the system is spread across teams, and nobody in the organization knows how to deploy the entire application.</span></span> <span data-ttu-id="b7bce-133">예를 들어 새 클러스터에 빠르게 배포해야 하는 경우 재해 복구 시나리오에서 무슨 일이 발생하나요?</span><span class="sxs-lookup"><span data-stu-id="b7bce-133">For example, what happens in a disaster recovery scenario, if you need to quickly deploy to a new cluster?</span></span>

## <a name="updating-services"></a><span data-ttu-id="b7bce-186">서비스 업데이트</span><span class="sxs-lookup"><span data-stu-id="b7bce-186">Updating services</span></span>

### <a name="canary-release"></a><span data-ttu-id="b7bce-220">카나리아 릴리스</span><span class="sxs-lookup"><span data-stu-id="b7bce-220">Canary release</span></span>

<span data-ttu-id="b7bce-221">카나리아 릴리스에서는 업데이트된 버전을 적은 수의 클라이언트로 롤아웃합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-221">In a canary release, you roll out an updated version to a small number of clients.</span></span> <span data-ttu-id="b7bce-222">그런 다음 모든 클라이언트에 롤아웃하기 전에 새로운 서비스의 동작을 모니터링합니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-222">Then you monitor the behavior of the new service before rolling it out to all clients.</span></span> <span data-ttu-id="b7bce-223">이를 통해 제어된 방식으로 느린 출시를 수행하고, 모든 고객이 영향을 받기 전에 실제 데이터와 스폿 문제를 확인할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="b7bce-223">This lets you do a slow rollout in a controlled fashion, observe real data, and spot problems before all customers are affected.</span></span>
