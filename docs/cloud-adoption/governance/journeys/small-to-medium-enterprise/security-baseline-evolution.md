# <a name="caf-small-to-medium-enterprise-security-baseline-evolution"></a>CAF: 중소 엔터프라이즈: 보안 기준 진화

이 문서에서는 보호된 데이터를 클라우드로 이동하는 것을 지원하는 보안 컨트롤을 추가하여 내러티브를 개선합니다.

## <a name="evolution-of-the-narrative"></a>내러티브의 개선

IT 및 비즈니스 리더십은 IT, 앱 개발 및 BI 팀에 의한 초기 단계 실험의 결과에 만족합니다. 이러한 실험의 실질적인 비즈니스 가치를 실현하려면 해당 팀은 보호된 데이터를 솔루션에 통합할 수 있어야 합니다. 이렇게 하면 회사 정책의 변경을 트리거하지만 보호된 데이터가 클라우드로 이동되기 전에 클라우드 거버넌스 구현의 개선이 필요합니다.

### <a name="evolution-of-the-cloud-governance-team"></a>클라우드 거버넌스 팀의 개선

지금까지 제공된 변화하는 내러티브 및 지원의 영향을 고려하면 클라우드 거버넌스 팀은 이제 달리 보입니다. 이 팀을 시작한 두 시스템 관리자는 이제 숙련된 클라우드 설계자로 보일 수 있습니다. 이 내러티브가 전개되면서 두 관리자에 대한 인식은 클라우드 보유자에서 클라우드 보호자 역할로 바뀝니다.

미묘한 차이지만 이 차이는 거버넌스 중점의 IT 문화권을 빌드할 경우 중요합니다. 클라우드 보유자는 혁신적인 클라우드 설계자가 만든 문제를 정리합니다. 이 두 역할에는 자연스러운 충돌과 반대되는 목표가 있습니다. 반면, 클라우드 보호자는 클라우드를 안전하게 유지할 수 있으므로 다른 클라우드 설계자는 거의 문제를 일으키지 않고 더 신속하게 이동할 수 있습니다. 또한 클라우드 보호자는 배포 및 채택을 가속화하는 템플릿 만드는 데 관여하며, 5개 클라우드 분야를 옹호할 뿐만 아니라 혁신을 가속화합니다.

### <a name="evolution-of-the-current-state"></a>현재 상태의 개선

이 내러티브의 초반에 애플리케이션 개발 팀은 여전히 개발/테스트 용량을 작업하고 있었으며, BI 팀은 실험 단계에 머물러 있었다. IT는 Prod 및 DR이라는 호스팅된 인프라 환경 2개에서 운영됐습니다.

그 이후로 거버넌스에 영향을 주는 다음의 몇 가지 사항이 변경되었습니다.

- 애플리케이션 개발 팀이 개선된 사용자 환경으로 클라우드 네이티브 애플리케이션을 배포하기 위해 CI/CD 파이프라인을 구현했습니다. 해당 앱은 아직 보호된 데이터와 상호 작용하지 않으므로 프로덕션이 준비가 되지 않았습니다.
- IT 팀 내의 비즈니스 인텔리전스 팀이 물류, 인벤토리 및 타사의 클라우드에서 데이터를 적극적으로 큐레이션합니다. 이 데이터는 비즈니스 프로세스를 작성할 수 있는 새로운 예측을 추진하는 데 사용되고 있습니다. 그러나 고객 및 금융 데이터를 데이터 플랫폼에 통합할 때까지는 이러한 예측 및 인사이트가 실행 가능하지 않습니다.
- IT 팀이 DR 데이터 센터의 사용을 중지하는 CIO 및 CFO의 계획을 진행하고 있습니다. DR 데이터 센터의 2,000개 자산 중 1,000개를 초과하는 자산이 사용 중지되거나 마이그레이션되었습니다.
- PII 및 금융 데이터에 관해 느슨하게 정의된 정책이 현대화되었습니다. 그러나 새 회사 정책은 관련 보안 및 거버넌스 정책의 구현 여부에 따르게 됩니다. 관련 팀의 작업은 계속 중단 상태입니다.

### <a name="evolution-of-the-future-state"></a>향후 상태의 개선

App Dev 팀과 BI 팀의 초기 실험 결과, 고객 환경 및 데이터 기반 결정 과정이 개선될 가능성이 있는 것으로 나타났습니다. 이 두 팀은 이러한 솔루션을 프로덕션 환경으로 배포하여 향후 18개월 동안 클라우드 도입 범위를 확장하려고 합니다.

남은 6개월 동안 클라우드 거버넌스 팀은 보안 및 거버넌스 요구 사항을 구현하여 클라우드 도입 팀이 해당 데이터 센터의 보호된 데이터를 마이그레이션할 수 있게 합니다.

현재와 미래의 상태가 변경되면 새로운 정책 설명이 필요한 새로운 위험에 노출됩니다.

## <a name="evolution-of-tangible-risks"></a>실질적인 위험의 개선

**데이터 침해**: 새 데이터 플랫폼을 도입할 경우 잠재적인 데이터 침해 관련 책임이 기본적으로 증가하게 됩니다. 이로 인해 클라우드 기술을 채택하는 기술자가 이 위험을 줄일 수 있는 솔루션을 구현할 책임이 증가했습니다. 기술자가 이러한 책임을 이행하도록 하려면 엄격한 보안 및 거버넌스 전략을 구현해야 합니다.

이 비즈니스 위험은 다음과 같은 몇 가지 기술적 위험으로 확장될 수 있습니다.

- 실수로 중요 업무용 앱이나 보호된 데이터를 배포할 수 있습니다.
- 암호화를 부적절하게 결정하여 보호된 데이터가 스토리지 중에 노출될 수 있습니다.
- 권한이 없는 사용자가 보호된 데이터에 액세스할 수 있습니다.
- 외부 침입으로 인해 보호된 데이터에 액세스될 수 있습니다.
- 외부 침입 또는 서비스 거부 공격으로 인해 비즈니스가 중단될 수 있습니다.
- 조직 또는 고용 상태가 변경되어 보호된 데이터에 무단 액세스가 허용될 수 있습니다.
- 새로운 악성 공격으로 인해 새로운 침입 또는 액세스 기회가 생길 수 있습니다.
- 일관성 없는 배포 프로세스로 인해 보안 허점이 발생하여 데이터가 유출되거나 중단될 수 있습니다.
- 구성 드리프트 또는 패치 누락으로 인해 의도하지 않은 보안 허점이 발생하여 데이터가 유출되거나 중단될 수 있습니다.

## <a name="evolution-of-the-policy-statements"></a>정책 문의 개선

정책을 다음과 같이 변경하면 새로운 위험을 완화하고 구현 과정을 안내할 수 있습니다. 항목이 매우 많은 것 같지만 실제로 이러한 정책을 도입하는 과정은 보기보다 쉬울 수 있습니다.

1. 배포된 모든 자산은 중요도 및 데이터 분류를 기준으로 분류되어야 합니다. 자산을 클라우드로 배포하기 전에 클라우드 거버넌스 팀과 애플리케이션 소유자가 분류를 검토해야 합니다.
2. 보호된 데이터를 저장하거나 액세스하는 애플리케이션은 이렇게 하지 않는 애플리케이션과 다른 방식으로 관리해야 합니다. 최소한 의도치 않게 보호된 데이터에 액세스하지 않도록 해당 애플리케이션을 분할해야 합니다.
3. 보호된 데이터는 미사용 시 모두 암호화해야 합니다.
4. 보호된 데이터를 포함하는 모든 세그먼트에서 권한이 상승된 사용 권한은 예외로 해야 합니다. 이러한 모든 예외는 클라우드 거버넌스 팀에서 기록되어 정기적으로 감사를 받습니다.
5. 보호된 데이터를 포함하는 네트워크 서브넷은 기타 모든 서브넷에서 격리해야 합니다. 보호된 데이터 서브넷 간의 네트워크 트래픽은 정기적으로 감사를 받습니다.
6. 공용 인터넷이나 데이터 센터를 통해 보호된 데이터를 포함하는 서브넷에 직접 액세스할 수 없습니다. 해당 서브넷에 액세스하려면 중간 서브넷을 통해 라우팅되어야 합니다. 해당 서브넷에 대한 모든 액세스는 패킷 검사 및 차단 기능을 수행할 수 있는 방화벽 솔루션을 거쳐야 합니다.
7. 거버넌스 도구는 보안 관리 팀에서 정의된 네트워크 구성 요구 사항을 감사하고 적용해야 합니다.
8. 거버넌스 도구는 VM 배포를 승인된 이미지로만 제한해야 합니다.
9. 가능한 경우 항상 노드 구성 관리에서 모든 게스트 운영 체제의 구성에 정책 요구 사항을 적용해야 합니다.
10. 거버넌스 도구는 모든 배포된 자산에 자동 업데이트가 사용되도록 적용해야 합니다. 위반은 운영 관리 팀에서 검토되고 작업 정책에 따라 수정돼야 합니다. 자동으로 업데이트되지 않는 자산은 IT 운영 팀에서 소유한 프로세스에 포함돼야 합니다.
11. 모든 중요 업무용 애플리케이션 또는 보호된 데이터용으로 새 구독이나 관리 그룹을 만들 경우 청사진이 적절하게 할당되도록 클라우드 거버넌스 팀의 검토가 필요합니다.
12. 중요 업무용 앱이나 보호된 데이터를 포함하는 모든 관리 그룹 또는 구독에는 최소 권한 액세스 모델이 적용됩니다.
13. 보안 팀에서 클라우드 배포에 영향을 줄 수 있는 추세 및 악용을 정기적으로 검토하여 클라우드에서 사용되는 보안 관리 도구에 업데이트를 제공해야 합니다.
14. 배포된 자산에 거버넌스를 지속적으로 적용하려면 클라우드 거버넌스 팀에서 배포 도구를 승인해야 합니다.
15. 정기적인 검토 및 감사를 위해서는 클라우드 거버넌스 팀이 액세스할 수 있는 중앙 리포지토리에서 배포 스크립트를 유지 관리해야 합니다.
16. 모든 자산에서 일관성을 유지하려면 거버넌스 프로세스에 배포 시점의 감사 내용과 정기적인 주기로 실행되는 감사 내용이 포함되어야 합니다.
17. 고객 인증이 필요한 모든 애플리케이션을 배포할 경우 내부 사용자용 기본 ID 공급 기업과 호환되는 승인된 ID 공급 기업을 사용해야 합니다.
18. 클라우드 거버넌스 프로세스에는 ID 관리 팀의 분기별 검토가 포함돼야 합니다. 이러한 검토를 통해 클라우드 자산 구성으로 방지해야 되는 악의적인 행위자 또는 사용 패턴을 쉽게 파악할 수 있습니다.

## <a name="evolution-of-the-best-practices"></a>모범 사례 개선

거버넌스 MVP 디자인을 개선하여 Azure Cost Management의 구현 및 새로운 Azure 정책을 포함합니다. 이러한 두 가지 디자인 변경을 통해 새 회사 정책 문이 실현됩니다.

1. 네트워킹 팀 및 IT 보안 팀에서는 네트워크 요구 사항을 정의합니다. 클라우드 거버넌스 팀은 대화를 지원합니다.
2. ID 팀 및 IT 보안 팀에서는 요구 사항을 정의하고 로컬 Active Directory 구현에 필요한 모든 변경을 수행합니다. 클라우드 거버넌스 팀에서는 변경 내용을 검토합니다.
3. 모든 관련 Azure Resource Manager 템플릿과 스크립팅된 구성을 저장하고 버전을 지정하려면 Azure DevOps에서 리포지토리를 만듭니다.
4. Azure Security Center 구현:
    1. 보호된 데이터 분류를 포함하는 모든 관리 그룹용으로 Azure Security Center를 구성합니다.
    2. 패치가 규정을 준수하도록 하려면 기본적으로 자동 프로비저닝을 사용으로 설정합니다.
    3. OS 보안 구성을 설정합니다. IT 보안 팀에서는 구성을 정의합니다.
    4. Security Center를 처음 사용할 경우 IT 보안 팀을 지원합니다. Security Center를 IT 보안 팀으로 전환하여 사용한다고 해도 액세스를 유지하여 거버넌스를 지속적으로 개선합니다.
    5. 구독 내의 Security Center 구성에 필요한 변경 내용을 반영하는 Resource Manager 템플릿을 만듭니다.
5. 모든 구독에 대해 Azure 정책을 업데이트합니다.
    1. 모든 관리 그룹과 구독에서 중요도 및 데이터 분류를 감사하고 적용하여 보호된 데이터 분류를 통해 모든 구독을 파악합니다.
    2. 승인된 이미지 사용만 감사하고 적용합니다.
6. 보호된 데이터 분류를 포함하는 모든 구독에 대해 Azure 정책을 업데이트합니다.
    1. 표준 Azure RBAC 역할의 사용만 감사하고 적용합니다.
    2. 개별 노드에서 미사용 상태의 모든 스토리지 계정과 파일에 대해 암호화를 감사하고 적용합니다.
    3. 모든 NIC 및 서브넷에 NSG의 적용을 감사하고 적용합니다. 네트워킹 팀 및 IT 보안 팀에서는 NSG를 정의합니다.
    4. 네트워크 인터페이스별 승인된 네트워크 서브넷 및 vNet 사용을 감사하고 적용합니다.
    5. 사용자 정의 라우팅 테이블의 제한을 감사하고 적용합니다.
    6. 다음과 같이 게스트 구성에 대해 기본 제공 정책을 적용합니다.
        1. Windows 웹 서버가 보안 통신 프로토콜을 사용 중인지 감사
        2. 암호 보안 설정이 Linux 및 Windows 머신 내부에 제대로 설정되었는지 감사
7. 방화벽 구성:
    1. 필요한 보안 요구 사항을 충족하는 Azure Firewall의 구성을 식별합니다. 또는 Azure와 호환되는 호환 가능한 타사 어플라이언스를 식별합니다.
    2. Resource Manager 템플릿을 만들어 필요한 구성으로 방화벽을 배포합니다.
8. Azure 청사진:
    1. `protected-data`라는 새 청사진을 만듭니다.
    2. 청사진에 방화벽 및 Azure Security Center 템플릿을 추가합니다.
    3. 보호된 데이터 구독용으로 새 정책을 추가합니다.
    4. 현재 보호된 데이터를 호스트하려는 모든 관리 그룹에 청사진을 게시합니다.
    5. 영향을 받는 각 구독에 기존 청사진 외에 새 청사진을 적용합니다.

## <a name="conclusion"></a>결론

위에 언급된 프로세스와 변경 내용을 거버넌스 MVP에 추가하면 보안 거버넌스와 연관된 대부분의 위험을 완화할 수 있습니다. 이러한 프로세스와 변경 내용은 데이터를 보호하는 데 필요한 네트워크, ID 및 보안 모니터링 도구를 추가합니다.
