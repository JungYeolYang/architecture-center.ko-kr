# 자동 확장
[!INCLUDE [header](../_includes/header.md)]

자동 확장은 응용 프로그램에 필요한 리소스를 동적으로 할당하여 런타임 비용을 최소화하면서 성능 요구사항과 SLA(서비스 수준 계약)는 충족시키는 프로세스입니다. 작업량이 증가함에 따라, 응용 프로그램이 제시간에 작업을 수행할 수 있도록 하려면 추가 리소스가 필요할 수 있습니다. 수요가 적어지면 리소스 할당이 해제되어, 비용은 최소화하고 성능은 계속 적절하게 유지되면서 SLA를 충족합니다. 
자동 확장은 클라우드 호스팅 환경의 탄력성을 활용하는 동시에 관리 오버헤드는 낮춰 줍니다. 운영자가 시스템 성능을 지속적으로 모니터링해야 하고 리소스 추가 또는 제거에 대한 결정을 내려야 할 필요성을 줄이기 때문입니다. 

> [!참고]
> 자동 확장은 컴퓨팅 리소스만이 아니라 응용 프로그램에서 사용하는 모든 리소스에도 적용됩니다. 예를 들어, 시스템이 메시지 대기열을 사용하여 정보를 보내고 받는 경우에는 규모가 커질 때 추가 대기열을 만들 수 있습니다. 
>
>

## 확장 유형
확장 형식은 일반적으로 다음 두 가지 중 하나에 속합니다. 

* **수직** (흔히 *업 앤 다운(up and down)* 확장이라고 함). 이 형식을 사용하려면 하드웨어를 수정(용량과 성능을 확장 또는 축소)하거나 적절한 용량과 성능을 갖춘 대체 하드웨어를 사용하여 솔루션을 다시 배포해야 합니다. 클라우드 환경에서 하드웨어 플랫폼은 보통 가상화된 환경입니다. 원래 하드웨어가 실질적으로 초과 프로비저닝된 경우를 제외하고, 초기 자본 비용이 있기 때문에 이 환경에서 수직 확장이 되면 보다 강력한 리소스를 프로비저닝하게 되고 이후에 시스템이 새로운 리소스로 이동하게 됩니다. 수직 확장은 흔히 파괴적인 프로세스여서 재배포 과정에서 시스템을 일시적으로 사용할 수 없습니다. 새 하드웨어를 프로비저닝하고 온라인 상태로 전환하는 과정에서 원래 시스템의 작동을 유지하는 것이 가능할 수도 있지만 이전 환경에서 새 환경으로 전환하는 동안 약간의 중단이 발생할 가능성이 큽니다. 자동 확장을 사용하여 수직 확장 전략을 구현하면 그럴 가능성이 적습니다.
* **수평** (흔히 *아웃 앤 인(out and in) 확장* 이라고 함). 이 형식을 사용하려면 추가 리소스나 더 적은 리소스, 즉 고성능 시스템이 아닌 일반적인 범용 리소스에 솔루션을 배포해야 합니다. 이 솔루션은 이러한 리소스가 프로비저닝되는 동안에도 중단 없이 계속 실행될 수 있습니다. 프로비저닝 프로세스가 완료되면 솔루션을 구성하는 요소의 사본을 이러한 추가 리소스에 배포하여 사용 가능하게 할 수 있습니다. 수요가 적어지면, 추가 리소스를 사용하는 요소가 완전히 종료된 후 추가 리소스를 회수할 수 있습니다. Microsoft Azur 등의 많은 클라우드 기반 시스템이 이 확장 형식의 자동화를 지원합니다.

## 자동 확장 전략 구현
자동 확장 전략을 구현하는 데는 일반적으로 다음과 같은 구성요소와 프로세스가 사용됩니다. 

* 응용 프로그램, 서비스, 인프라 레벨의 계측기 및 모니터링 시스템. 이러한 시스템은 응답 시간, 대기열 길이, CPU 사용률, 메모리 사용량 등의 주요 메트릭을 캡처합니다.
* 사전 정의된 시스템 임계값이나 일정에 대해 모니터링된 확장 요소를 평가할 수 있고 확장 여부를 결정하는 의사 결정 논리.
* 리소스의 프로비저닝 또는 디프로비저닝 등 시스템 확장과 관련된 작업을 수행하는 구성요소.
* 기능이 예상대로 작동되는지 확인하기 위한 자동 확장 전략의 테스트, 모니터링 및 조정.

Azure 같은 클라우드 기반 환경 대부분에서는 일반적인 시나리오를 다루는 기본 제공 자동 확장 메커니즘을 제공합니다. 사용 중인 환경이나 서비스에서 필요한 자동 확장 기능을 제공하지 않는 경우, 또는 해당 기능 이상의 매우 뛰어난 자동 확장이 필요한 경우, 사용자 지정 구현을 해야 할 수 있습니다. 이 사용자 지정 구현을 사용하여 운영 및 시스템 메트릭을 수집하고 그것을 분석하여 관련 데이터를 식별한 후 그에 따라 리소스를 확장합니다. 

## Azure 솔루션용 자동 확장 구성
Azure 솔루션용 자동 확장을 구성하는 옵션은 다음과 같이 몇 가지가 있습니다. 

* **Azure 자동 확장** 일정을 기반으로 가장 일반적인 확장 시나리오를 지원하고, 선택적으로는 런타임 메트릭(예: 프로세서 사용률, 대기열 길이, 기본 제공 카운터 또는 사용자 지정 카운터)을 기반으로 트리거된 확장 작업을 지원합니다. 사용자는 Azure 포털을 사용하여 솔루션용의 간단한 자동 확장 정책을 쉽고 빠르게 구성할 수 있습니다. 보다 세밀한 제어를 위해 [Azure Service Management REST API](https://msdn.microsoft.com/library/azure/ee460799.aspx) 또는 [Azure Resource Manager REST API](https://msdn.microsoft.com//library/azure/dn790568.aspx)를 사용할 수 있습니다. [Azure Monitoring Service Management Library](http://www.nuget.org/packages/Microsoft.WindowsAzure.Management.Monitoring)와 [Microsoft Insights Library](https://www.nuget.org/packages/Microsoft.Azure.Insights/)(미리보기 참고)는 다른 리소스에서 메트릭을 수집하고 REST API를 활용하여 자동 확장을 수행할 수 있게 해주는 SDK입니다. Azure Resource Manager 지원을 사용할 수 없는 리소스의 경우, 또는 Azure Cloud Services를 사용 중인 경우에는 Service Management REST API를 자동 확장에 사용할 수 있습니다. 그 외의 모든 경우에는 Azure Resource Manager를 사용합니다.
* **사용자 지정 솔루션** 응용 프로그램의 계측기와 Azure의 관리 기능을 기반으로 하는 사용자 지정 솔루션이 유용할 수 있습니다. 예를 들어, Azure 진단 기능 또는 응용 프로그램의 기타 계측기를 사용할 수 있으며, 이와 함께 사용자 지정 코드를 사용하여 응용 프로그램의 메트릭을 지속적으로 모니터링하고 내보낼 수 있습니다. 이러한 메트릭에서 작동하는 사용자 지정 규칙을 가질 수 있으며 자동 확장을 트리거하기 위해 Service Management REST API 또는 Resource Manager REST API를 활용할 수 있습니다. 확장 작업을 트리거하기 위한 메트릭은 기본 제공 카운터나 사용자 지정 카운터, 또는 응용 프로그램 내에서 구현하는 기타 계측기가 될 수 있습니다. 그러나 사용자 지정 솔루션은 구현하기가 쉽지 않으며 이전 방법 중 어느 것도 사용자 요구사항을 충족하지 못하는 경우에만 이 솔루션을 고려해야 합니다. [Autoscaling Application Block](http://msdn.microsoft.com/library/hh680892%28v=pandp.50%29.aspx)은 이 방법을 사용합니다.
* **제3자 서비스** [Paraleap AzureWatch](http://www.paraleap.com/AzureWatch) 등의 제3자 서비스를 사용하면 일정, 서비스 부하 및 시스템 성능 표시기, 사용자 지정 규칙, 서로 다른 규칙 유형의 조합을 기반으로 솔루션을 확장할 수 있습니다.

자동 확장 솔루션을 선택할 때는 다음 사항을 고려하십시오. 

* 요구사항을 충족할 수 있다면 플랫폼에서 기본 제공되는 자동 확장 기능을 사용하십시오. 그렇지 않은 경우에는 더 복잡한 확장 기능이 정말 필요한지에 대해 신중히 고려하십시오. 추가 요구사항의 예로는 추가적인 제어 세분화, 확장을 위한 트리거 이벤트 감지, 구독을 통한 확장, 다른 유형의 리소스 확장 등을 들 수 있습니다.
* 예정된 자동 확장에만 의존하여 충분한 정확도로 응용 프로그램의 부하를 예측할 수 있는지를 고려하십시오(예상되는 최대 수요를 충족시키기 위해 인스턴스 추가 및 제거). 이렇게 할 수 없는 경우에는, 런타임에 수집된 메트릭을 기반으로 하는 반응형 자동 확장을 사용하여 응용 프로그램이 예기치 않은 수요 변화에 대응할 수 있도록 하십시오. 일반적으로 다음과 같은 방법을 결합할 수 있습니다. 예를 들어, 응용 프로그램이 가장 많이 사용되는 시간을 기준으로 컴퓨팅, 저장소, 대기열 등의 리소스를 추가하는 전략을 만듭니다. 이렇게 하면 새 인스턴스 시작 시 지연이 발생하지 않고 필요시 용량을 사용할 수 있습니다. 또한 예정된 각 규칙의 경우, 해당 기간 동안 반응형 자동 확장을 허용하는 메트릭을 정의하여, 지속적이지만 예기치 않은 최대 수요를 응용 프로그램이 처리할 수 있도록 합니다.
* 메트릭과 용량 요구사항 간의 관계를 이해하는 것은 어려울 때가 많습니다. 응용 프로그램을 처음 배포했을 때는 특히 어렵습니다. 처음에는 추가 용량을 약간 프로비저닝한 후, 자동 확장 규칙을 모니터링 및 조정하여 용량을 실제 부하에 가깝게 만들어야 합니다.

### Azure 자동 확장 사용
자동 확장을 사용하면 솔루션에서 스케일 아웃과 스케일 인 옵션을 구성할 수 있습니다. 자동 확장은 Azure Cloud Services 웹 역할 및 작업자 역할, Azure Mobile Services, Azure App Service 웹 앱 기능의 인스턴스를 자동으로 추가 및 제거할 수 있습니다. Azure Virtual Machines의 인스턴스를 시작하고 중지하여 자동 확장을 활성화할 수 있습니다. Azure 자동 확장 전략에는 다음 두 가지 요소의 집합이 포함되어 있습니다. 

* 일정 기반 자동 확장은 추가 인스턴스를 최대 예상 사용량과 일치시킬 수 있고 최대 사용량 시간이 경과하면 규모를 축소시킬 수 있습니다. 이렇게 하면 시스템이 부하에 반응할 때까지 기다리지 않고도 충분한 인스턴스가 실행되도록 할 수 있습니다.
* 메트릭 기반 자동 확장은 마지막 1시간 동안의 평균 CPU 사용률, 또는 솔루션이 Azure 저장소나 Azure Service Bus 대기열에서 처리 중인 메시지 백 로그 등의 요소에 반응합니다. 그러면 응용 프로그램이 예정된 자동 확장 규칙과 별개로 반응하여 계획되지 않았거나 예기치 않은 수요 변화에 대응할 수 있습니다.

자동 확장을 사용할 때는 다음 사항을 고려하십시오. 

* 자동 확장 전략은 예정된 확장과 메트릭 기반 확장을 모두 결합합니다. 사용자는 두 가지 유형의 서비스 규칙을 모두 지정할 수 있습니다.
* 자동 확장 규칙을 구성한 후, 지속적으로 응용 프로그램의 성능을 모니터링해야 합니다. 이 모니터링 결과를 사용하여 필요시 시스템을 확장하는 방법을 조정하면 됩니다. 그러나 자동 확장은 즉각적인 프로세스가 아니라는 점을 명심하십시오. 지정된 임계값을 초과하는(또는 하회하는) 평균 CPU 사용률 등의 메트릭에 반응하는 데는 시간이 걸립니다.
* 측정된 트리거 속성(예: CPU 사용률 또는 대기열 길이)에 기반한 탐지 메커니즘을 사용하는 자동 확장 규칙은 즉각적인 값을 사용하는 것이 아니라 시간이 지남에 따라 집계된 값을 사용하여 자동 확장 작업을 트리거합니다. 기본적으로는, 값의 평균이 집계됩니다. 이렇게 하면 시스템이 너무 빨리 반응하거나 급격한 변동이 발생하는 것을 방지할 수 있습니다. 또한 자동으로 시작된 새 인스턴스가 실행 모드로 들어갈 시간을 허용하므로, 새 인스턴스가 시작되는 동안 추가 자동 확장 작업이 발생되지 않게 해줍니다. Azure Cloud Services와 Azure Virtual Machines의 경우, 기본 집계 시간이 45분이므로 수요 급증에 대응하여 메트릭이 자동 확장을 트리거하는 데는 이 시간이 모두 필요할 수 있습니다. SDK를 사용하여 집계 시간을 변경할 수 있지만, 25분 미만은 예기치 않은 결과를 초래할 수 있습니다(자세한 내용은 [Azure Monitoring Services Management Library에서 CPU 백분율에 대한 클라우드 서비스 자동 확장](http://rickrainey.com/2013/12/15/auto-scaling-cloud-services-on-cpu-percentage-with-the-windows-azure-monitoring-services-management-library/) 참조). 웹 앱의 경우에는 평균 시간이 훨씬 짧으며, 평균 트리거 측정을 변경한 후 약 5분 이내에 새 인스턴스를 사용할 수 있습니다.
* 웹 포털이 아닌 SDK를 사용하여 자동 확장을 구성하는 경우에는 규칙이 활성화되는 동안 보다 자세한 일정을 지정할 수 있습니다. 또한 자체 메트릭을 생성하여 자동 확장 규칙에서 사용할 수 있으며, 이때 기존 메트릭은 사용할 수도 있고 사용하지 않을 수도 있습니다. 예를 들어, 초당 요청 수나 평균 메모리 가용성 등의 대체 카운터를 사용할 수도 있고 특정 비즈니스 프로세스를 측정하는 사용자 지정 카운터를 사용할 수도 있습니다.
* Azure Virtual Machines를 자동 확장할 때는 자동 확장을 시작할 수 있는 최대 수와 동일한 수의 가상 머신 인스턴스를 배포해야 합니다. 이러한 인스턴스는 동일한 가용성 집합의 일부여야 합니다. 가상 머신 자동 확장 메커니즘은 가상 머신의 인스턴스를 생성하거나 삭제하지 않습니다. 대신, 사용자가 구성하는 자동 확장 규칙이 이러한 인스턴스의 적절한 수를 시작하고 중지하게 됩니다. 자세한 내용은 [웹 역할, 작업자 역할 또는 가상 머신을 실행하는 응용 프로그램의 자동 확장](/azure/cloud-services/cloud-services-how-to-scale/)을 참조하십시오.
* 구독 최대값에 도달하거나 시작 중에 오류가 발생하는 등의 이유로 새 인스턴스를 시작할 수 없는 경우, 포털에서 자동 확장 작업이 성공했음을 표시할 수 있습니다. 그러나, 포털에 표시되는 후속 **ChangeDeploymentConfiguration** 이벤트에서는 서비스 시작이 요청되었다는 것만 표시하고 서비스 시작이 성공적으로 완료되었음을 나타내는 이벤트는 표시하지 않습니다.
* 사용자는 웹 포털 UI를 사용하여 SQL 데이터베이스 인스턴스, 대기열 등의 리소스를 컴퓨팅 서비스 인스턴스에 연결할 수 있습니다. 이렇게 하면 연결된 각 리소스에 대한 별도의 수동 및 자동 확장 구성 옵션에 보다 쉽게 액세스할 수 있습니다. 자세한 내용은 [클라우드 서비스 관리 방법](/azure/cloud-services/cloud-services-how-to-manage) 및 [응용 프로그램 확장 방법](/azure/cloud-services/cloud-services-how-to-scale/)에 있는 "리소스를 클라우드 서비스에 연결하는 방법"을 참조하십시오.
* 정책과 규칙을 여러 개 구성하면 서로 충돌할 수 있습니다. 자동 확장은 다음과 같은 충돌 해결 규칙을 사용하여 충분한 인스턴스가 항상 실행되도록 합니다.
  * 스케일 아웃 작업은 항상 스케일 인 작업보다 우선합니다.
  * 스케일 아웃 작업이 충돌하면, 인스턴스 수를 가장 많이 증가시키는 규칙이 우선합니다.
  * 스케일 인 작업이 충돌하면, 인스턴스 수를 가장 적게 감소시키는 규칙이 우선합니다.

## 자동 확장 구현을 위한 응용 프로그램 설계 고려사항
자동 확장은 즉각적인 해결책이 아닙니다. 단순히 시스템에 리소스를 추가하거나 실행 프로세스의 인스턴스 수를 늘린다고 해서 시스템 성능이 개선될 것이라는 보장은 없습니다. 자동 확장 전략을 수립할 때는 다음 사항을 고려하십시오. 

* 시스템이 수평적으로 확장 가능하도록 설계해야 합니다. 인스턴스 선호도에 대한 가정을 피하십시오. 코드가 항상 프로세스의 특정 인스턴스에서 실행되어야 하도록 솔루션을 설계하지 마십시오. 클라우드 서비스나 웹 사이트를 수평으로 확장할 때 동일한 소스에서 오는 일련의 요청이 항상 동일한 인스턴스로 라우팅된다고 가정하지 마십시오. 같은 이유로, 응용 프로그램에서 오는 일련의 요청이 항상 동일한 서비스 인스턴스로 라우팅되어야 하는 상황을 피하기 위해 서비스를 상태 비저장으로 설계하십시오. 대기열의 메시지를 읽고 처리하는 서비스를 설계할 때, 특정 메시지를 처리하는 서비스 인스턴스가 정해져 있다는 가정을 하지 마십시오. 자동 확장은 대기열 길이가 늘어남에 따라 추가 서비스 인스턴스를 시작할 수 있습니다.[경쟁 소비자 패턴](http://msdn.microsoft.com/library/dn568101.aspx)은 이 시나리오를 처리하는 방법을 설명합니다.
* 솔루션이 장기 실행 작업을 구현하는 경우, 이 작업이 스케일 아웃과 스케일 인을 모두 지원하도록 설계하십시오. 주의를 기울이지 않으면 시스템이 축소될 때 이러한 작업이 프로세스 인스턴스의 완전한 종료를 방해할 수 있습니다. 또는 프로세스를 강제로 종료할 경우 데이터가 손실될 수 있습니다. 가장 좋은 방법은 장기 실행 작업을 리팩터링하고 처리 작업을 세분화하여 더 작은 개별 덩어리가 되도록 하는 것입니다. [파이프 및 필터 패턴](http://msdn.microsoft.com/library/dn568100.aspx)은 이렇게 하는 방법의 예를 보여줍니다.
* 또는 정기적인 간격으로 작업에 대한 상태 정보를 기록하는 검사점 메커니즘을 구현하고, 이 작업을 실행하는 프로세스의 모든 인스턴스로 액세스할 수 있는 지속형 저장소에 이 상태를 저장하면 됩니다. 이런 방식으로 하여, 프로세스가 종료되면 수행 중이던 작업을 다른 인스턴스를 사용하여 마지막 검사점에서 다시 시작할 수 있습니다.
* 배경 작업이 클라우드 서비스 호스팅 응용 프로그램의 작업자 역할에서와 같이 별도의 컴퓨팅 인스턴스에서 실행되면 다른 확장 정책을 사용하여 응용 프로그램의 다른 부분을 확장해야 할 수 있습니다. 예를 들어, 배경 컴퓨팅 인스턴스의 수를 늘리지 않고 추가 UI(사용자 인터페이스) 컴퓨팅 인스턴스를 배포해야 할 수도 있고 그 반대일 수도 있습니다. 사용자가 여러 수준의 서비스를 제공하는 경우(예: 기본 서비스와 고급 서비스 패키지), SLA를 충족하기 위해 기본 서비스 패키지에 대한 컴퓨팅 리소스보다는 고급 서비스 패키지에 대한 컴퓨팅 리소스를 보다 적극적으로 확장해야 할 수 있습니다.
* UI와 배경 컴퓨팅 인스턴스가 자동 확장 전략의 기준으로 통신하도록 대기열의 길이를 사용하는 것을 고려해 보십시오. 이것은 배경 작업의 처리 용량과 현재 부하 간의 불균형이나 차이를 나타내는 가장 좋은 지표입니다.
* 시간당 배치된 주문 수나 복잡한 트랜잭션의 평균 실행 시간 등, 비즈니스 프로세스를 측정하는 카운터에 자동 확장 전략을 적용하는 경우에는 이러한 유형의 카운터에서 나온 결과와 실제 컴퓨팅 용량 요구사항 간의 관계를 완벽하게 이해해야 합니다. 비즈니스 프로세스 카운터의 변경에 대응하여 둘 이상의 구성요소 또는 컴퓨팅 단위를 확장해야 할 수 있습니다.  
* 시스템이 과도하게 확장되는 것을 방지하고 수천 개의 인스턴스 실행으로 인한 비용을 피하려면 자동으로 추가할 수 있는 인스턴스의 최대 수를 제한하는 것이 좋습니다. 대부분의 자동 확장 메커니즘은 규칙의 최소 및 최대 인스턴스 수를 지정할 수 있게 해줍니다. 또한, 최대 인스턴스 수가 배포되고 시스템이 여전히 과부하 상태이면 시스템이 제공하는 기능을 적절하게 저하시키는 방법을 고려하십시오.
* 급격히 증가하는 작업량을 처리하는 데 자동 확장이 가장 적합한 메커니즘은 아닐 수 있음을 명심하십시오. 새로운 서비스 인스턴스를 프로비저닝하고 시작하거나 시스템에 리소스를 추가하는 데는 시간이 걸리는데, 이러한 추가 리소스를 사용할 수 있게 될 때쯤 최대 수요는 사라질 수 있습니다. 이 시나리오에서는 서비스를 제한하는 것이 좋을 수 있습니다. 자세한 내용은 [스로틀링 패턴](http://msdn.microsoft.com/library/dn589798.aspx)을 참조하십시오.
* 반대로, 작업량이 급격히 변동할 때 모든 요청을 처리할 용량이 필요하고 이때 비용이 주요 요인이 아닌 경우에는 추가 인스턴스를 더 빨리 시작하는 공격적인 자동 확장 전략을 사용하는 것을 고려해 보십시오. 부하가 예상되기 전에 최대 부하를 충족하기 위해 충분한 인스턴스 수를 시작하는 예정된 정책을 사용할 수도 있습니다.
* 자동 확장 메커니즘은 자동 확장 프로세스를 모니터링하고 각 자동 확장 이벤트의 상세 정보를 기록해야 합니다(이벤트 촉발 요인, 추가되거나 제거된 리소스 및 그 시점). 사용자 지정 자동 확장 메커니즘을 생성할 때는 이 기능이 통합되어 있는지 확인하십시오. 자동 확장 전략의 효율성을 측정하고 필요한 경우 조정할 수 있도록 정보를 분석하십시오. 사용 패턴이 더 명확해지면 단기적으로, 비즈니스가 확장되거나 응용 프로그램의 요구사항이 발전하면 장기적으로 조정할 수 있습니다. 응용 프로그램이 자동 확장을 위해 정의된 상한선에 도달하면, 필요시 수동으로 추가 리소스를 시작할 수 있는 운영자에게 메커니즘이 경고를 보낼 수도 있습니다. 이러한 상황에서 작업자는 작업량이 줄어든 후 이 리소스를 수동으로 제거해야 할 수도 있음을 유의하십시오.

## 관련 패턴과 지침
다음 패턴과 지침은 자동 확장을 할 때 사용자 시나리오에 적용될 수도 있습니다. 

* [스로틀링 패턴](http://msdn.microsoft.com/library/dn589798.aspx). 이 패턴은 수요 증가로 리소스에 부하가 극심할 때 응용 프로그램이 SLA를 충족하고 계속 기능할 수 있게 하는 방법을 설명합니다. 스로틀링은 자동 확장과 함께 사용되어 시스템이 확장되는 동안 과부하를 방지할 수 있습니다.
* [경쟁 소비자 패턴](http://msdn.microsoft.com/library/dn568101.aspx). 이 패턴은 모든 응용 프로그램 인스턴스의 메시지를 처리할 수 있는 서비스 인스턴스의 풀을 구현하는 방법을 설명합니다. 서비스 인스턴스를 시작하고 중지하여 예상되는 작업량과 일치시키는 데 자동 확장을 사용할 수 있습니다. 이 방법을 통해 시스템은 여러 메시지를 동시에 처리하여 처리량을 최적화하고, 확장성과 가용성을 향상시키고, 작업량의 균형을 유지할 수 있습니다.
* [계측기 및 원격 측정 장치 지침](http://msdn.microsoft.com/library/dn589775.aspx). 계측기 및 원격 측정 장치는 자동 확장 프로세스를 수행할 수 있는 정보를 수집하는 데 매우 중요합니다.
